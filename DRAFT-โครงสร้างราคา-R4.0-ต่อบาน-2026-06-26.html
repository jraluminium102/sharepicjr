🏗️ โครงสร้างราคา R4.0 — ปรับตรงไหน + ราคาต่อบานเข้ายังไง

ตอบ 3 เรื่อง: (1) พี่กรอกราคาตรงไหน · (2) โปรเจคใหม่หรือทำต่อ · (3) ออกแบบราคาต่อบาน

🔒 ดราฟวางแผน · อ่านโค้ดจริงมาประกอบ · ยังไม่แตะ index.html

1️⃣ พี่กรอก/ปรับราคาตรงไหน?

ราคาทั้งระบบรวมอยู่ที่ RATES{ } ก้อนเดียว (index.html บรรทัด 576) — จุดแก้เดียว สินค้าทุกตัวชี้มาที่นี่

// ตารางราคากลาง — แก้ตัวเลขที่นี่ที่เดียว ทั้งเว็บเปลี่ยนตาม const RATES = { SMS: [[2.0,2.3,6500],[2.3,3.5,6000], ... [12,9999,4000]], EURO: [[2.0,2.3,7200],[2.3,3.5,6600], ... ], // └ช่วงล่าง └ช่วงบน └เรต/ตร.ม. (← เลขที่พี่ปรับ) }
📖 อ่านยังไง: [2.0, 2.3, 6500] = บานพื้นที่ 2.0–2.3 ตร.ม. คิด 6,500 บาท/ตร.ม.
R4.0 = แค่เปลี่ยนชุดเลขนี้ (ทุน/ตร.ม. × 2 ปัดร้อย) โครงเดิมไม่ขยับ → เสี่ยงต่ำสุด งานไม่พัง

2️⃣ เปิดโปรเจคใหม่ หรือทำต่อในนี้?

ประเด็น🟢 ทำต่อโปรเจคเดิม (แนะนำ)🔴 เปิดโปรเจค/เว็บใหม่
ความเร็วเร็ว — แก้แค่ตัวเลขช้า — ทำ engine/ใบ/ฟอร์มใหม่หมด
งานพังเสี่ยงต่ำ + มี golden snapshot กันเสี่ยงสูง — ของที่เทสแล้วต้องเทสซ้ำหมด
เฟิร์นตรวจเทียบเก่า→ใหม่ในที่เดียวต้องเรียนระบบใหม่
อนาคตปรับราคากดเปลี่ยนเลข RATES จบเท่ากัน แต่ลงทุนทำใหม่ก่อน
✅ คำตอบ: ทำต่อในโปรเจคเดิม — engine เดียว เปลี่ยนแค่ "ตัวเลข + เพิ่ม field ต่อบาน" ไม่ต้อง fork เว็บใหม่
💡 ส่วน "แชท" ค่อยเปิดใหม่ได้ (แชทนี้ยาว) — แต่ "โค้ด/เว็บ" ใช้ตัวเดิม · R4.0 ทำเป็น "ชุดเลขที่สลับเข้าได้" ปลอดภัยกว่าสร้างใหม่

3️⃣ ราคาต่อบาน — ออกแบบให้มีผลยังไง

สถานะตอนนี้: เครื่องคิดราคา "ต่อบาน" มีอยู่แล้วบางชนิด (ของจริงในโค้ด):

  • บานเฟี้ยม → unit_rate คิดต่อบาน + ขั้นต่ำ 15,000/บาน
  • บานเปิดยูโร → ceLinear คิดต่อบาน (Linear)
  • บานเปลือยประตู → per_panel: 8000 บวกต่อบาน
  • บานยก → คิดต่อ "ชุด" (บังคับ 2 บาน) + บานเพิ่ม

ช่องว่าง: บานเลื่อน/ติดตาย/อีกหลายตัว = คิดแค่ พื้นที่อย่างเดียว — บาน 2 ตร.ม. เท่ากันคิดเท่ากันหมด ไม่ว่าจะ 2 บานหรือ 4 บาน (แต่ทุนจริงต่างกัน เพราะบานเยอะ = เฟรม/ล้อ/มือจับเยอะ)

📌 ในไฟล์ถอดทุน 4.0 มี "จำนวนบาน" เป็นตัวแปร → R4.0 ควรให้จำนวนบานมีผล มี 3 ทางเลือก:

ทาง A — เรตต่อ ตร.ม. (เดิม) + บวก "ต่อบาน" แยกแนะนำ ✅

เพิ่ม field ใหม่ในแต่ละสินค้า panelAdd = ค่าเพิ่มต่อบาน · ราคาคิดแบบ:

ราคา = (พื้นที่ × เรต/ตร.ม.) + (จำนวนบาน × panelAdd)
ตัวอย่าง บานเลื่อน SMS 4 บาน พื้นที่ 6 ตร.ม. · panelAdd = 800/บาน
= (6 × 4,100) + (4 × 800) = 24,600 + 3,200 = 27,800 · ถ้า 2 บานพื้นที่เท่ากัน = 24,600 + 1,600 = 26,200 (ถูกกว่าเพราะบานน้อย ✔)
ข้อดี: โครงเดิมอยู่ครบ · เพิ่มแค่ field เดียว · จำนวนบานมีผลทันที · กดปรับ 2 ตัวเลข (เรต + ต่อบาน) · golden กันพังได้
ข้อควรระวัง: ต้องตั้ง panelAdd ต่อชนิด (แต่ดึงจากไฟล์ทุน 4.0 ได้เลย)

ทาง B — ตารางเรต 2 มิติ (พื้นที่ × จำนวนบาน)กลาง

เปลี่ยน RATES ให้มีเรตแยกตามจำนวนบานด้วย เช่น เรต 2 บาน / 3 บาน / 4 บาน คนละชุด

ราคา = พื้นที่ × เรต[ช่วงพื้นที่][จำนวนบาน]
ข้อดี: แม่นกับทุนจริงสุด (ทุนต่อบานไม่เชิงเส้น)
ข้อเสีย: ตารางใหญ่ขึ้น 3–4 เท่า · กรอก/ตรวจเยอะ · engine ต้องแก้ลึก · เสี่ยงพังสูงกว่า

ทาง C — คิดทุนจริงต่อบานทั้งหมด (rebuild)หนัก

เลิกเรตต่อ ตร.ม. ทั้งหมด → คิดจาก น้ำหนักอลู × ราคา/กก. × จำนวนเส้น + กระจก + ค่าแรง ต่อบานจริง (ตามไฟล์ 4.0 ตรงๆ)

ข้อดี: แม่นที่สุด · กดเปลี่ยนน้ำหนัก/ราคาอลู ราคาขยับเอง (เป้าหมายปลายทาง)
ข้อเสีย: รื้อ engine ใหญ่ · เสี่ยงพังสูงสุด · ใช้เวลานาน · เฟิร์นตรวจยาก
✅ แนะนำเริ่ม "ทาง A" — ได้ "จำนวนบานมีผล" ทันที โดยโครงเว็บไม่พัง · เป็นบันไดไปทาง C ในอนาคต (ค่อยๆ ขยับ)
เพราะมติคือ markup 100% เป็นฐาน อยู่แล้ว → ทาง A = เรต/ตร.ม.(×2) + ต่อบาน(×2) ก็ครบเป้า เร็ว แม่น ไม่พัง

⏭️ ถ้าโอเค — ลำดับทำ (ปรับจากแผนเดิม)

1 ดึงจากไฟล์ 4.0: ทุน/ตร.ม. (×2) + ทุนต่อบาน (×2) ทุกกลุ่ม

2 ดราฟตารางเทียบ R3.9 → R4.0 (มีคอลัมน์ "ต่อบาน" เพิ่ม) → เฟิร์นตรวจ

3 ใบสั่ง dev: เปลี่ยนเลข RATES + เพิ่ม field panelAdd + แก้ engine จุดเดียว (บวกต่อบาน) + golden snapshot
