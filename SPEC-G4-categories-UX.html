1. ปัญหาที่พบ — keyword miss ทำให้ตกหมวดอื่นๆ

ต้นเหตุ: groupGHOpts (บรรทัด ~1750) ตรวจ textContent ของ label แต่ keyword ใน CATS ไม่ตรงกับ text จริงในโค้ด:
ช่อง (class)label จริงใน render (บรรทัด ~2621)keyword ใน CATSผล
.o-depth "ลึก(ม.)" ความลึก ← ไม่ match ตกอื่นๆ
.o-shelves "จำนวนชั้น (ว่าง=อัตโนมัติ)" จำนวนบาน / ชนิดชั้น ← ไม่ match ตกอื่นๆ
.o-backwall "+ผนังหลัง" ไม่มี keyword ใดในหมวดหลัก ← ไม่ match ตกอื่นๆ
.o-cabtype "ประเภท" — match keyword ประเภท ประเภท ✓ เข้า ชนิด/การเปิด
.o-cabdoors "จำนวนบานหน้าตู้" — match keyword จำนวนบาน จำนวนบาน ✓ เข้า ชนิด/การเปิด
.o-shelfmat "ชนิดชั้น" — match keyword ชนิดชั้น ชนิดชั้น ✓ เข้า ชนิด/การเปิด

ผลลัพธ์ปัจจุบัน: ตู้อลูมี 6 ช่อง → 3 เข้าหมวด "ชนิด/การเปิด" ปนกับ option บาน/ประตูอื่น · 3 ตกหมวด "อื่นๆ" ปนกับครอบวงกบ/หมายเหตุ

2. โครงหมวดใหม่สำหรับ G4 — ตู้อลู

หลักการ: ของถามบ่อย/กำหนดราคา → เปิด default · เสริม/นานๆใช้ → พับ default · เป้าหมายกด 1-2 tap ถึงช่องหลัก

ลำดับการกรอก (บนลงล่าง)

1 ประเภทตู้ ชิป
2 จำนวนบานหน้าตู้ ชิป
3 ชนิดชั้น ชิป
4 ลึก(ม.) auto
5 จำนวนชั้น auto
6 +ผนังหลัง checkbox
7 ครอบวงกบอลู (ชิป · เสริม)
8 มีคาดตาราง (checkbox · เสริม)
9 งานเสริม / รื้อของเดิม
10 หมายเหตุ / OPTION

ข้อ 1-6 = "รายละเอียดตู้" เปิด default · ข้อ 7-10 = "เสริม / หมายเหตุ" พับ default

หมวด "มือจับ/ล็อค" และ "มุ้ง" ไม่มี item ของตู้ → groupGHOpts จะข้ามให้อัตโนมัติ (buckets ว่าง)

3. แนวทางที่เลือก — (ก) เพิ่มหมวด "📦 รายละเอียดตู้" ใน CATS

เลือกแนว (ก): เพิ่ม cat ใหม่เข้า CATS array ก่อนหมวด "อื่นๆ" + เติม keyword ให้ match label จริง
เหตุผล: idempotent, ปลอดภัย, ไม่ต้องเพิ่ม DOM พิเศษ, ใช้ mechanism เดิม 100%, dev เข้าใจง่าย
ทำไมไม่ใช้แนว (ข) — เขียน groupGHOpts ใหม่สำหรับ cabinet โดยเฉพาะ:
ข้อเสีย = logic แยก 2 สาย ทำให้ maintain ยาก + ถ้าเพิ่มสินค้าใหม่ต้องแก้ 2 ที่ · แนว (ก) แค่เพิ่ม keyword row เดียว

ห้ามลบ 'ประเภท', 'จำนวนบาน' ออกจากหมวด 🚪 — สินค้าอื่นพึ่งคำเหล่านี้อยู่:
  • G1 บานเลื่อน: label "จำนวนบาน" → ต้องอยู่ใน 🚪 · ถ้าลบออก → หลุดเป็น "อื่นๆ" ทันที (regression)
  • G3 หลังคา: label "ประเภท..." → ต้องอยู่ใน 🚪 (match ด้วย substring)
วิธีที่ปลอดภัย: ให้ label ของตู้ใช้คำที่ ยาวกว่า / เฉพาะกว่า เช่น "ประเภทตู้" และ "จำนวนบานหน้าตู้" แล้ววาง 📦 ไว้ก่อน 🚪 ใน array — loop เจอ 📦 match ก่อน break ทันที ตู้ไม่หลุดไป 🚪 · สินค้าอื่นไม่มีคำเหล่านี้ใน label → ไม่หลุดเข้า 📦 เลย

ข้อสังเกต keyword "ชนิดชั้น": คำนี้เฉพาะตู้จริง (สินค้าอื่นไม่มี) การไว้ทั้งใน 📦 และ 🚪 ไม่ใช่ปัญหา — 📦 อยู่ก่อน จะ intercept ก่อนเสมอ

3.2 ปรับ label ของ o-cabtype ในบล็อก render (บรรทัด ~2621)

ปัจจุบัน label = "ประเภท" → match keyword ประเภท ในหมวด "ชนิด/การเปิด" ก่อน · แก้เป็น "ประเภทตู้" ให้ match หมวดใหม่แทน (เพราะ "ประเภทตู้" ยาวกว่า ไม่ match ประเภท ใน 🚪 ด้วย substring)

// บรรทัด ~2621 — if(p.cabinet){ ... }
// เดิม:
h+='<label class="opt">ประเภท <select class="o-cabtype">...</select></label>'
// แก้เป็น:
h+='<label class="opt">ประเภทตู้ <select class="o-cabtype">...</select></label>'
// หรือถ้าทำเป็นชิปแล้ว (ตาม SPEC-G4-formux) ให้ใช้ label "ประเภทตู้" ในชิป-wrapper
// ⚠ "ประเภทตู้" match keyword 'ประเภทตู้' ใน 📦 ✓
// ⚠ "ประเภทตู้" ไม่ match keyword 'ประเภท' ใน 🚪 เพราะ indexOf('ประเภท') ≥0 แต่ 📦 อยู่ก่อน → break แล้ว

5. วิธีซ่อน 3 ออปชั่นที่ตัดออก — Pattern โค้ด (ปลอดภัย)

กฎเหล็ก: ห้ามลบ element — ให้ซ่อนด้วย conditional ใน string h ก่อน append (render ไม่ออก) หรือ style="display:none" หลัง render · ห้าม remove node ที่ readItem อาจ querySelector อยู่

5.1 ซ่อน ฝังรางยู (.o-uchannel) บรรทัด ~2680

เดิม: if(!['หลังคา','ฝ้า-ผนัง'].includes(p.cat)) → เพิ่ม cabinet/future_tech เข้า condition:

// บรรทัด ~2680 — แก้ condition
// เดิม:
if(!['หลังคา','ฝ้า-ผนัง'].includes(p.cat)){
  h+='<label...>ฝังรางยู...</label>';
}

// แก้เป็น — เพิ่มเช็ค p.cabinet และ p.future_tech:
if(!['หลังคา','ฝ้า-ผนัง'].includes(p.cat) && !p.cabinet && !p.future_tech){
  h+='<label...>ฝังรางยู...</label>';
}

5.2 ซ่อน ดรอปพื้น (.o-dfm) บรรทัด ~2691

ดรอปพื้นอยู่ใน template string เดียวกับครอบวงกบ (บรรทัด ~2691) โดย concatenate ต่อกัน:

// บรรทัด ~2691 — ปัจจุบันมีทั้ง ครอบวงกบ และ ดรอปพื้นในบรรทัดเดียว
// logic: if(!areaOnly && !p.zipscreen){ h += ครอบวงกบ + ดรอปพื้น }

// แก้: แยก ดรอปพื้น ออกมา ซ่อนเฉพาะ p.cabinet/p.future_tech
if(!areaOnly && !p.zipscreen){
  // ครอบวงกบ — แสดงทุกสินค้า (รวมตู้)
  h+=`<label class="opt" style="width:100%;">ครอบวงกบอลู ...</label>`;
  // ดรอปพื้น — ซ่อนเฉพาะตู้/ฝาตู้
  if(!p.cabinet && !p.future_tech){
    h+=`<label class="opt">ดรอปพื้น(ม.) <input type="number" class="o-dfm" ...></label>`;
  }
}

// readItem ยังทำงาน: บรรทัด ~2851 คือ const df=d.querySelector('.o-dfm'); if(df)optSel.dfm=df.value;
// เมื่อ element ไม่ถูก render → querySelector คืน null → if(df) ไม่รัน → optSel.dfm ไม่ถูก set (ปลอดภัย)

5.3 ซ่อน ประเภทประตู/หน้าต่าง (.i-type chip + wrapper) บรรทัด ~2211

.i-type อยู่ใน addItem template ตายตัว (ทุกสินค้า) — ซ่อนหลัง render โดยใช้ buildItemOpts ซึ่งรู้ p แล้ว:

// ในฟังก์ชัน buildItemOpts(d, p) — หลัง box.innerHTML=h; (บรรทัด ~2705)
// ซ่อน .i-type wrapper เมื่อเป็น cabinet หรือ future_tech
if(p.cabinet || p.future_tech){
  const _tw = d.querySelector('.i-type');
  if(_tw){
    // หา wrapper ระดับ div.full (parent ของ chip-group + select)
    const _twrap = _tw.closest('div.full');
    if(_twrap) _twrap.style.display = 'none';
  }
}

// ✅ ปลอดภัย: select .i-type ยังอยู่ใน DOM (display:none ไม่ remove)
// readItem บรรทัด ~2888: const _mit=d.querySelector('.i-type'); optSel.itype=(_mit&&_mit.value)||'door'
// → querySelector หา display:none ได้ → value ยังอ่านได้ · default 'door' ไม่กระทบ calcUnit ตู้
